# Реалізація GOF патернів проєктування



Цей проект демонструє реалізацію 9 основних патернів проєктування Gang of Four.



## Як запустити

1. Встановіть залежності: `npm install`

2. Запустіть демонстрацію: `npx ts-node examples/run-all.ts`



## Реалізовані патерни

### Породжувальні

- **Singleton**: Гарантує один екземпляр класу.

	Проблема: Потрібно гарантувати, що у класі є лише один екземпляр (наприклад, підключення до БД).

	Анти-приклад: Створення нового підключення щоразу, що вичерпує ліміти сокетів.

- **Factory Method**: Делегує створення об'єктів підкласам.

	Проблема: Клас не знає заздалегідь, об'єкти яких підкласів йому потрібно створювати.

	Анти-приклад: Використання if-else або switch по всьому коду для створення об'єктів різних типів.

- **Builder**: Дозволяє створювати складні об'єкти крок за кроком.

	Проблема: Конструктор з 10+ параметрами (Telescoping Constructor).

	Анти-приклад: new Car('BMW', 'X5', 2023, true, false, 'Red', ...) — легко переплутати порядок.



### Структурні

- **Adapter**: Дозволяє об'єктам з несумісними інтерфейсами працювати разом.

	Проблема: Несумісність інтерфейсів сторонніх бібліотек або старого коду.

	Анти-приклад: Переписування всього старого коду під новий інтерфейс.

- **Facade**: Надає простий інтерфейс до складної системи.

	Проблема: Складна підсистема з купою класів, якими важко керувати.

	Анти-приклад: Клієнтський код вручну ініціалізує 10 об'єктів підсистеми для однієї дії.

- **Decorator**: Динамічно додає нову функціональність.
	
	Проблема: Потрібно динамічно додавати функціонал об'єктам без створення підкласів для кожної комбінації.

	Анти-приклад: Створення класів CoffeeWithMilk, CoffeeWithSugar, CoffeeWithMilkAndSugar.



### Поведінкові

- **Strategy**: Визначає сімейство алгоритмів та робить їх взаємозамінними.
	
	Проблема: Багато if-else для вибору алгоритму (наприклад, різні методи оплати).

	Анти-приклад: Величезний метод pay(method: string) з логікою всередині.

- **Observer**: Дозволяє об'єктам підписуватися на події інших об'єктів.
	
	Проблема: Потрібно сповіщати декілька об'єктів про зміну стану одного.

	Анти-приклад: Постійне опитування (polling) об'єкта "чи змінився ти?".

- **Command**: Перетворює запит на об'єкт.
	
	Проблема: Потрібно параметризувати об'єкти операціями, ставити їх у чергу або скасовувати.
